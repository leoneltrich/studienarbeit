\section{Methodology}
This chapter describes the methodological approach used to design, implement, and evaluate the proposed local-first command option retrieval system. The methodology consists of a literature review, a requirements-driven system design process, an implementation and model selection phase, technical benchmarking, and a user study. Together, these steps provide the basis for evaluating both the technical feasibility of the approach and its perceived usefulness in command-line workflows.

\subsection{System Design}
The system design process began by defining functional and non-functional requirements for the application, in addition to the relevant environmental constraints. These requirements and constraints provided the foundation for the subsequent architecture selection and modeling phases.

To determine which architectural pattern best fit the requirements, different patterns were systematically analyzed based on qualitative criteria. After selecting the architectural pattern, the system was modeled using the C4 standard~\cite{brown2026c4}. This provided a high-level overview of the system, its components, and its boundaries. To provide a more detailed view of the system behavior, BPMN 2.0 ** ref BMPN ** style diagrams were created to describe the execution flow of the system and its different functionalities.

Finally, the internal and external interfaces of the system were defined, and the technologies most suitable for implementing the system were selected. The technology selection was based on qualitative and quantitative criteria. This concluded the groundwork for the system implementation.

\subsection{Model Selection}

The selection of the retrieval model began with the definition of an initial candidate set. This set was derived from the ranked list of models with fewer than one billion parameters on the MTEB English v2 leaderboard~\cite{mteb_leaderboard} **source for MTEB benchmark**. As this benchmark provides a focused overview of models that perform strongly across a range of English-language embedding tasks, it served as an appropriate basis for the initial selection process. The initial candidate set was then refined to account for task suitability and architectural diversity, as described in Chapter~\ref{sec:comp_emb_models}. The aim of this step was to obtain a reduced set of at least three suitable candidate models for the subsequent evaluation stage. In addition, multiple quantized variants of the selected models were tested whenever they satisfied the available memory constraints, in order to account for potential performance differences between quantization levels of the same base model.

In a second step, the remaining candidate models were compared using two retrieval tasks based on the validation set described in Chapter~\ref{sec:statistical_benchmarking}.

The first task focused on tool retrieval. For this purpose, an example catalog was created for the \texttt{mv} and \texttt{cp} tools (see Appendix~\ref{appendix:tool_catalog_examples}). Using these catalogs, each sample query from the validation set was embedded and compared against the corresponding tool descriptions. The dot product between query and tool embeddings was then used to determine which of the two candidate tools received the higher similarity score. Each model variant was assigned an accuracy score corresponding to the percentage of cases in which the correct tool was ranked above the incorrect one.

The second task focused on option retrieval and split into multiple steps. First, a baseline test was run utilizing an Otsu cutoff **source** to select the subset of the most relevant options. This baseline was evaluated on the raw sample queries from the validation set. During this stage, the following metrics were collected:

**sources for those:**
\begin{enumerate}
    \item Precision: The ratio of correctly predicted options to the total predicted options. 
    \begin{equation}
        \text{Precision} = \frac{\text{TP}}{\text{TP} + \text{FP}}
    \end{equation}
    
    \item Recall: The proportion of options actually required that were correctly identified by the model. 
    \begin{equation}
        \text{Recall} = \frac{\text{TP}}{\text{TP} + \text{FN}}
    \end{equation}
    
    \item F1-Score: The harmonic mean of Precision and Recall, providing a balanced metric.
    \begin{equation}
        \text{F}_1 = 2 \times \frac{\text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}
    \end{equation}
    
    \item \ac{RSF}: A metric that measures how much the total number of predicted options deviates from the expected number, focusing purely on volume rather than whether the specific choices are correct.
    \begin{equation}
        \text{RSF} = \begin{cases}
           1.0 & \text{if } |P| = 0 \text{ and } |E| = 0 \\
           \max\left(\frac{|P|}{|E|}, \frac{|E|}{|P|}\right) & \text{if } |P| > 0 \text{ and } |E| > 0 \\
           \big||E| - |P|\big| + 1.0 & \text{otherwise}
        \end{cases}
    \end{equation}
    
    \item Recall/\ac{RSF}: The ratio of Recall to the \ac{RSF} ($\frac{\text{Recall}}{\text{RSF}}$), used to evaluate a model's efficiency by penalizing accurate recall that relies on massive over-prediction.
\end{enumerate}

In addition to the baseline, several variations of the retrieval setup were evaluated in an exhaustive grid search, in which every possible combination of the selected variations was tested. For each model, the best-performing configuration across all tested combinations was retained, and the resulting best scores were used to construct a final comparison table. This table formed the basis for selecting the model that performed best on the available data.

The following three variations were tested individually and in all possible combinations:

\begin{enumerate}
    \item Appending the target tool to the input query in the format \textit{"<query> Tool: <tool>"}, since this information is already available at the stage where command options are retrieved.
    \item AAdding a title to the option description. Instead of embedding only the description itself, the input was reformulated as \textit{"title: <tool> <flag> | text: <description>"}. 
    \item Appending catalog keywords to the option description in the format \textit{"<description> Keywords: <keywords>"} in order to test whether keyword augmentation improves retrieval performance.
\end{enumerate}

Because the project targets an efficient retrieval system, additional efficiency-related metrics were collected alongside the retrieval quality metrics. These included peak memory consumption during embedding, model load time, and average throughput in tokens per second. To ensure accurate and isolated measurements, each benchmark run was executed in a fresh subprocess, preventing interference from previous model evaluations. Peak memory usage was obtained directly from \texttt{/proc/self/status} via the \texttt{VmHWM} field, which reports the process high-water mark~\cite{linux_proc_status}. Model load time was measured with a timer started immediately before model loading and stopped directly after context initialization, thereby excluding unrelated benchmark setup overhead. Throughput was calculated by dividing the total number of processed tokens by the cumulative time spent exclusively in the decoding stage, while excluding tokenization and all post-processing steps.


\subsection{Benchmarking}
After the system design and implementation methodology had been established, the next methodological step was to define how the system would be evaluated technically. This section is divided into statistical benchmarking, which focuses on dataset generation and utilization, and performance benchmarking, which outlines the benchmarking environment used.

\subsubsection{Statistical Benchmarking} \label{sec:statistical_benchmarking}
To allow for a systematic evaluation of different parameter configurations, a dataset was created. The dataset was intended to provide a set of sample queries that a user might submit to the system in order to achieve a specific task. For this purpose, ten sample categories were first defined (Appendix **X**), five of which used the "mv" tool and five of which used the "cp" tool. Each category consisted of a destination command and a description of the function performed by that command. A destination command was defined as a full command, including the tool name, options, flags, and any arguments required to perform the action described.

To obtain a sufficiently large and varied set of sample queries for benchmarking, the large language model Gemini 3.1 Flash Lite was used. The exact prompt was optimized for creating versatile samples and is documented in Appendix **X**. The generated dataset consisted of 1219 sample queries. These queries were almost equally distributed across the categories, with only two deviations from the expected 120 samples per category.
The first deviation occurred in category ID 1. This was caused by an initial test run that created 20 additional samples, resulting in a total of 140 samples for that category. The second deviation occurred in category ID 4. It resulted from a redundancy protection feature that was implemented to prevent multiple instances of identical sample queries. This feature converts all characters to lowercase, strips excess whitespace, and removes punctuation. It then stores the normalized string together with the original query, provided that no identical normalized string already exists in the database. This mechanism prevented one duplicate sample in category ID 4, resulting in a total of 119 sample queries for that category.

To prevent bias toward the dataset, a stratified 50/50 split was created, resulting in a validation set and a test set. Since the dataset contained an uneven number of records, the validation set consisted of 610 samples and the test set consisted of 609 samples. During development, only the validation set was used to tune the system. The test set was used only at the end, after the system had been finalized, for the final system evaluation.

To ensure that the final benchmark was statistically reliable, the test set size ($N = 609$) was evaluated using the standard margin of error for proportions. Since this work is a Proof of Concept (PoC), a 90\% confidence level ($Z = 1.645$) was chosen. Based on this sample size and confidence threshold, the maximum theoretical margin of error is $\pm3.3\%$ under the condition of maximum variance ($p = 0.5$). The margin of error ($e$) for a population proportion is calculated as:

\begin{equation}
e = Z \cdot \sqrt{\frac{p(1-p)}{N}} = 1.645 \cdot \sqrt{\frac{p(1-p)}{609}}
\end{equation}

Here, $Z$ is the z-score for a 90\% confidence level, $N$ is the sample size, and $p$ represents the empirical success rate recorded during the benchmark.

\subsubsection{Performance Benchmarking}
Since the system follows a local-first approach, benchmarking must not be limited to the expected success rate alone. Resource usage and execution speed also need to be taken into account. Therefore, the execution environment was defined in table~\ref{tab}. The VM described there had several standard performance-related configuration options enabled manually to utilize the provided hardware as effectively as possible. The measures taken were:

\begin{enumerate}
	\item CPU set to host mode
	\item Hard disk enabled SSD emulation and discard option
\end{enumerate}

Most other configuration options remained at the Proxmox defaults applied during virtual machine creation, with one exception. To obtain performance closer to that of enterprise SSDs or bare-metal SSD storage, ZFS synchronous write handling was disabled using the command \textit{zfs set sync=disabled rpool/data/vm500-disk-0}. With this setting, ZFS may acknowledge synchronous write requests before the data has been durably committed to storage. This can increase apparent write performance, but it also reduces protection against data loss in the event of a crash or power failure. Therefore, the measured write speeds may be higher than those achievable by storage systems operating with standard synchronous-write guarantees.

\subsection{User Study}
In addition to the technical benchmarking, a user study was conducted to evaluate the perceived usefulness of the tool in realistic command-line workflows. While the benchmarking focused on measurable technical characteristics, the user study was intended to assess how participants experienced the tool in terms of accessibility, speed, and workflow support.

The goal of this project was to contribute to improving users' workflows when using the terminal, regardless of whether they were novice, intermediate, or advanced CLI users. The user study therefore evaluated the perceived effectiveness of the tool and assessed whether it supported participants when working with the CLI.
For the user study, participants with a wide spectrum of skill levels were selected. This was intended to reduce bias in the results. For example, more advanced users might have been able to solve tasks more easily due to their existing expertise, while less experienced users might have perceived the tool differently due to their limited CLI understanding. To support participant classification, a pre-study survey was created to divide participants into novice, intermediate, and expert categories.

\subsubsection{Pre-study Evaluation}
The pre-study evaluation consisted of 10 questions of varying difficulty, designed to assess participants' general knowledge of the CLI. The questions were compiled based on widely used operations, ranging from moving and locating files to understanding paths, managing permissions, and using commands (see Appendix **insert ref**). During the user study, the questions were shuffled randomly to improve the reliability of the survey and to reduce the likelihood of participants sharing answers.

The survey questions were written as multiple-choice questions, with four possible answers and an "I don't know" option for each question. The latter option was included to discourage random guessing and reduce distortion in the results. Asking specific knowledge-based questions instead of asking participants to estimate their own skill level is considered more suitable for accurately assessing user competence **SOURCE?**. Originally, the survey was planned to use written text responses **What "survey instrument"?**, but multiple-choice questions were selected instead. This reduced cognitive load, increased participant engagement, reduced ambiguity in the responses, and simplified grading.

Each correct answer added one point to a participant's score, and the skill levels were determined using the following score intervals:
\begin{itemize}
	\item 0 - 3: Beginner
	\item 4 - 7: Intermediate
	\item 8 - 10: Advanced
\end{itemize}

\subsubsection{User Evaluation}

After the pre-study evaluation, participants were asked to interact with the developed tool in a controlled evaluation environment. To give participants the opportunity to experience the tool and provide feedback, an evaluation platform was created. The platform consisted of a sandboxed testing environment and a catalog of tasks and questions. After receiving a short introduction to the tool (Appendix **insert ref**), participants were presented with three tasks involving the "mv" and "cp" utilities. These tasks allowed more proficient users to compare the tool's results with their own prior knowledge.

Subsequently, participants completed three additional tasks requiring the use of a custom, poorly documented tool with ambiguous option names. This was intended to demonstrate how the tool could support users when working with a proprietary command-line application they had not encountered before.

Participants were not required to successfully complete a task before moving on to the next one. However, returning to previous tasks was not possible, as the solution to each task was shown after proceeding. To allow participants to verify whether the tool had produced correct results, a verification command was provided for each task. This command could be used during the task to check whether it had been completed successfully. Since the survey focused on gathering qualitative feedback, task success was not recorded. However, the time taken by each participant to proceed to the next task was measured.
The time spent on each task was primarily measured using a WebSocket connection. This made it possible to track when the connection was established and when it was closed, for example when a participant closed the browser or the evaluation platform tab. In such cases, the timer could be paused until the participant returned to the task. However, the WebSocket connection remained active if the participant merely switched tabs or windows. To account for this, standard JavaScript APIs were used to track the tab state and send a pause signal to the backend when necessary.

To prevent participants from manipulating the system, escaping the sandbox, or interfering with other participants' environments, securely configured containers were used. The containers were executed with gVisor **insert ref** as the runtime. gVisor provides an additional isolation layer by running containers in their own sandboxed environment, reducing reliance on the shared host kernel and thereby increasing isolation. In addition, network and internet access were disabled for the sandboxes to prevent inter-container communication.

After completing the tasks, participants answered a total of twelve questions. The first nine questions were multiple-choice questions, while the remaining three required free-text responses.
