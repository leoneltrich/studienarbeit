\subsection{Benchmarking}
This section is split into the statistical benchmarking which, focuses on dataset generation and utilization, and the performance benchmarking, which talks about the benchmarking environment used.

\subsubsection{Statistical Benchamrking}
To allow for a founded evaluation of different parameter configurations we created a dataset. The dataset aimed at providing a set of sample queries that a user might give to the system to achieve a task. For this we first defined a total of ten sample categories (Appendix **X**), five of them utilizing the "mv"-tool and another five the "cp"-tool. Each category consists of a destination command and a description of what that command does. A destination command was defined as a full command, including the tool name, the options, the flags and any arguments required to achieve the action described in the description. 

Because of the vast sample size required to make a statistically valid claim we utilized the large language model Gemini 3.1 Flash Lite. The exact prompt used was optimized for creating versatile samples and is documented in appendix **X**. The generated dataset totaled 1219 sample queries. These queries are almost equally distributed across categories, with only two divergances from the expected 120 samples. 
The first divergence can be found for the  category with the ID 1. This is due to an initial test run that created 20 additional samples, resulting in a total of 140 sampels for that category. The second divergance can be found in the category with ID 4. It is a result of a redundancy protection feature we implemented, to prevent multiple instances of identical sample queries. The feature converts all characters to lowercase, strips all excess whitespaces and removes any punctuation. It then stores this normalized string along with the original query, provided no instance of the normalized string exists in the database yet. This mechanism prevented one duplicate sample for category 4, leading to a total of 119 sample queries for that category. 

To prevent bias toward the dataset, we decided to create a stratified 50/50 split resulting in a test and a validation set. Since the dataset has an uneven number of records the validation set consists of 610 samples and the test set of 609 samples. During the development process only the validation set was used to tune the system. The test set was used only at the very end, on the finished system, for the system evaluation. 

To ensure the final benchmark is statistically reliable, we evaluated our test set size ($N = 609$) using the standard margin of error for proportions. Since this is a Proof of Concept (PoC), a 90\% confidence level ($Z = 1.645$) was chosen. Based on this sample size and confidence threshold, the maximum theoretical margin of error is $\pm3.3\%$ under the condition of a maximum variance ($p = 0.5$). The margin of error ($e$) for a population proportion is calculated as:

\begin{equation}
e = Z \cdot \sqrt{\frac{p(1-p)}{N}} = 1.645 \cdot \sqrt{\frac{p(1-p)}{609}}
\end{equation}

With $Z$ being the z-score for 90\% confidence, $N$ being the sample size, and $p$ representing the empirical success rate recorded during the benchmark.

\subsubsection{Performance Benchmarking}
Since the system utilizes a local first approach we can not only limit our benchmarking to the expected success rate. The ressource usage and speed of the system also need to be taken into account. Therefore we defined the execution environment in table~\ref{tab:env_constraints}. The VM described had several standard performance measures manually enabled to utilize the provided hardware as effectively as possible. The measures taken are:

\begin{enumerate}
    \item CPU set to host mode
    \item Hard Disk enabled SSD emulation and discard option
\end{enumerate}

Most other configuration options remain in at the default set by Proxmox upon machine creation, with one exception. To get performance comparable to enterprise SSDs or an SSD in a bare metal system, we disabled ZFS write confirmation using the command \textit{zfs set sync=disabled rpool/data/vm500-disk-0}. Whilst the utilization of a Proxmox Virtual Machine for the benchmarking enabled us to achieve results applicable to real world server environments, it must be considered that this specific performance measure may cause write speeds even exceeding enterprise SSDs that do not have the confirmations turned off.

\subsection{System Design}

The systems design process started by defining the requirements for the application. In addition to functional and non-functional requiremnts, environment constraints were defined. This provided the the base for the following architecture pattern decision and modeling phases. For the architecture pattern decision different architecture patterns were systematically analysed. Based on qualitative aspects it was evaluated which architecture best fits the requirements. After the architecture pattern was chosen the system was modeled using the C4 standard~\cite{brown2026c4}. This provided a high level overview of the system, it's components and it's boundries. To gain a more detailled perspective on how the system works BPMN2.0 ** ref BMPN ** style diagrams were created to describe the execution flow of the system and it's different functionalities. Ultimately the internal and external interfaces of the system were defined and the technologies best suited to implement the system were chosen. To choose the best technologies for the system, different candidates were evaluated based on qualitative and quantitative measures. This concluded the ground work for the systems implementation.

\subsection{User Study}
The goal of this project was to contribute to improving a user’s workflow when using the terminal, regardless of whether they were novices, intermediate, or advanced CLI users. A goal of this study was to evaluate the perceived effectiveness of the tool by users, to assess whether it succeeded at helping the user through using the CLI, in terms of speed and accessibility.

For the user study, users with a wide spectrum of skill levels were selected. This was intended to avoid overall bias in the results, as, for example, more advanced users might have been able to resolve tasks with greater ease due to their existing expertise, while less experienced users might have found the tool less convenient due to their lack of CLI understanding. For the selection, a pre-study survey was created to divide the users into novice, intermediate, and expert categories.

\subsubsection{Pre-study Evaluation}

The pre-study evaluation was composed of 10 questions, of varying difficulty, that aimed to assess a user's general knowledge of the CLI. The questions were compiled based on widely used operations, ranging from moving and locating files, to the understanding of paths, managing permissions, and using commands (see appendix **insert ref**). During the user study, the questions were shuffled randomly to make the user survey as accurate as possible, thereby preventing users from potentially sharing answers.

The survey questions were written as multiple choice, with 4 possible answers each, and an "I don't know" option for each question. The latter option was implemented to prevent users from guessing randomly and skewing the results. Asking specific questions instead of asking users to estimate their skills is considered best for an accurate portrayal of user capacities **SOURCE?**. Originally, the survey instrument called for written text responses **What "survey instrument"?**, but it was determined that multiple choice would not only reduce cognitive load and maximize user engagement, but it would reduce ambiguity in the responses and allow for easier grading.

Lastly, each correct answer added one point to a user's score, and the levels were determined by the following score intervals:

\begin{itemize}
    \item 0 - 3: Beginner
    \item 4 - 7: Intermediate
    \item 8 - 10: Advanced
\end{itemize}

\subsubsection{User Evaluation}

To give the users a chance to experience the developed tool and give feedback on it,
we created a platform consisting of a sandboxed testing environment and a catalog of tasks and questions. After getting a short introduction to the tool (appendix **insert ref**) the users were presented with three tasks invloving the "mv" and "cp" tools. This particularly allowed more proficient users to compare the results of our tool with their own pre-existing knowledge. Consecutively, a set of three tasks was presented to the users, where the use of a custom, poorly documented tool, with ambiguous option names was required. This was to show the users how the tool might help them to use a proprietary tool they had never seen before. 

Users were not required to successfully complete a task to move on to the next one. However, they could not go back to a previous task, because they were provided with the solution to the previous task after moving on to the next one. To give users a way to verify, wether the tool yielded correct results, a verification command was provided for each task, that the users could use during the tasks to check, if they had successfully completed them. We did not track wether the users actually solved the tasks, since the survey was focussed on gathering qualitative feedback from the users. We did however track, how long it took each user to move on to the next task respectively.

The time spent per task, was generally measured using a websocket connection. Like this we could track the begin of the connection and the exact time it was closed, which would mean the user closed their browser or the tab the evaluation platform was running in. That allowed us to pause the timer until the user returned back to the task. However, the websocket connection would remain established, if the user only changed tabs or windows on their maschine. To also track this and pause the timer appropriately, we utilized standard Javascript APIs to track the state of the tab and send a pause signal to the backend if required.

To prevent users from manipulating the system, breaking out of the sandbox or messing with other users sandboxes we utilized securely configured containers. To secure the containers we used gVisor **insert ref** as a runtime, which runs the containers in their own micro \acp{VM} preventing a shared host kernel and thereby guranteeing an increased level of isolation. Forthermore network and internet access were disabled for the sandboxes to prevent any kind of inter-container communication.

After finishing the tasks the users were provided with a total of twelve questions, with the first nine of them being multiple choice and the remaining three requiring free text responses.
