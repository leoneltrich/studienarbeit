\section{Comparison Of Embedding Models}\label{sec:comp_emb_models}

For the project only embedding modells of the bi-encoder family were compared. Models of the cross-encoder family were not taken into account. The reason for this is the fundamentally different way the two work. While for a bi-encoder model the embeddings of a target description can be calculated and stored **source**, providing a significant boost in retrieval, this is not possible for cross-encoder models. That is, due to the way data is processed using cross-encoder models. The calculated similarity score in this case, is based on the combined input of the target description and the user query **source**. This makes cross-encoders more accurate, but at the same time slower than bi-encoders **source**, leading to the decision to focus on a bi-encoder model as the embedding engine in this project.

\subsubsection{Excluded Models}

While models such as \texttt{IEITYuan/Yuan-embedding-2.0-en}, \texttt{infgrad/Jasper-Token-Compression-600M}, and \texttt{KaLM-embedding-multilingual-mini-instruct-v2.5} rank highly on the leaderboard, they were deliberately excluded from our evaluation to maintain architectural diversity and mitigate structural redundancy.

\texttt{IEITYuan/Yuan-embedding-2.0-en} and \texttt{infgrad/Jasper-Token-Compression-600M} are both directly derived from \texttt{Qwen3-Embedding-0.6B} **source**, \texttt{KaLM-embedding-multilingual-mini-instruct-v2.5} is built on a Qwen2 backbone **source**. Including all three in the evaluation would therefore increase representation of the broader Qwen family and make the comparison less focused on genuinely independent embedding designs.

\subsubsection{Selected Models}

Firstly, we selected two Jina models for comparison. However, rather than using the higher-ranked Jina-v5 text variants listed on the benchmark, we chose the \texttt{small} and \texttt{nano} retrieval variants. In this project, the model must map a natural-language user query to short descriptions of command-line tools and their options, making the task an asymmetric query-document retrieval problem. Jina explicitly distinguishes retrieval from text-matching and provides task-specific adapters for these settings **source**. Accordingly, the \texttt{small} and \texttt{nano} retrieval variants were selected instead of their respective text equivalents, because they are designed for retrieval-oriented semantic search, whereas text-matching is intended for symmetric similarity tasks such as paraphrase or duplicate detection **source**. This ensures that the Jina models are evaluated in the configuration most appropriate for the command-retrieval task rather than being chosen solely on the basis of aggregate benchmark rank.

Google's \texttt{embeddinggemma-300m} was selected because it adds a smaller, independent comparison point to the evaluation while remaining competitive in the benchmark ranking. At 300M parameters, it offers a substantially lighter alternative to the larger models in the comparison, making it useful for assessing whether a more compact embedding model can still perform adequately on the command-retrieval task. This also improves the overall evaluation design by broadening the comparison beyond Qwen-related and Jina-based choices.

Despite the stronger leaderboard performance of some Qwen-derived variants, \texttt{Qwen/Qwen3-Embedding-0.6B} was selected as the primary baseline because it provides the clearest and most interpretable reference point within that model family. Yuan is explicitly built on \texttt{Qwen3-Embedding-0.6B}, Jasper is a downstream derivative of the same line, and KaLM is built on a Qwen2 backbone, so using \texttt{Qwen/Qwen3-Embedding-0.6B} as the primary representative of that family helps avoid repeatedly sampling closely related variants with additional downstream modifications. This choice preserves analytical clarity by separating the base Qwen embedding architecture from later fine-tuning and architectural adaptations, allowing the comparison to focus more directly on differences between model families rather than refinements within a single lineage.

In order to optimize results for each of the models, different modifications have been applied to the user queries and descriptions ingested.

\subsubsection{jina-v5-retrieval Models (small/nano)}
For the Jina-v5-retrieval models, the primary task-specific modification applied in this comparison was to prepend user queries with \textit{"Query: "} and tool and option descriptions with \textit{"Document: "}, following Jina's retrieval setup **source**.

\subsubsection{embeddinggemma-300m}
For \texttt{embeddinggemma-300m}, a task type must be specified so that the model can optimize embeddings for the intended retrieval setting **source**. In this evaluation, the task type \textit{"search result"} was used and prepended to each query together with the prefix \textit{"query: "}, yielding the format \textit{"task: search result | query: <query>"}. For the embedding of tool and option descriptions, a title-text structure was used in the form \textit{"title: <title> | text: <description>"} **source**. The title was defined as either the tool name or the combination of tool name and option, depending on the embedding target, and the text field contained the corresponding description.

\subsubsection{Qwen3-Embedding-0.6B}
For \texttt{Qwen3-Embedding-0.6B} custom prompt can be added before the query. While this allows for versatile testing, it makes some of th results highly circumstancial based on the prompt used. While one prompt might lead to good scores another prompt might have yielded worse ones. We therefore compared a total of three prompts for each, retrieving the tool and retrieving the options. For retriving the tool we compared the following prompts using the Q8 quantized version of the \texttt{Qwen3-Embedding-0.6B} model:

\begin{enumerate}
    \item Given a user command query, retrieve the matching command-line tool description. (Accuracy: 84.26\%)
    \item Given a command description, find the matching command name. (Accuracy: 82.95\%)
    \item Given a command, retrieve the corresponding tool description. (Accuracy: 83.28\%)
\end{enumerate}

The accuracy was in this case measured as the percentage of times, the model scored the correct tool higher then the incorrect one. The test was conducted using the validation set described in chapter~\ref{sec:statistical_benchmarking}. All prompts perform very similarly, however the first one got the highest score and was hence used in the further evaluation.

Using the same quantized model, the following prompt options were compared for retrieving the correct command options:

\begin{enumerate}
    \item Given a user command query, retrieve all matching command-line option descriptions.
    \item Given a user request, find the option flags that match the description.
    \item Retrieve the command-line flags that match the user request.
\end{enumerate}
