\subsection{Architectural Pattern}
\label{sec:architecture_philosophy}

The design of the system is fundamentally driven by the non-functional requirements governing latency, memory footprint, and modularity. Specifically, the system must process natural language queries in under 1000 ms (NFR-1) within a 1024 MB memory limit (NFR-3), while ensuring that the core domain logic remains strictly decoupled from volatile \ac{I/O} operations, allowing both the persistent storage backend (NFR-6) and the specific search and matching algorithms (NFR-7) to be seamlessly replaced.

To satisfy these constraints, a traditional Layered (N-Tier) Architecture was discarded. While a layered approach neatly categorizes concerns into different tiers (e.g. presentation, business and data), it introduces a transitive top-down dependency structure \cite{richards2020fundamentals}. As Richards and Ford assert, layers in this architecture remain isolated only if the contracts between them remain unchanged. However, in a standard layered model, the data access layer and external libraries define these contracts. Consequently, the core business logic is transitively coupled to the paradigms of the infrastructure. If the local storage technology or the specific text-matching algorithms were fundamentally replaced, the cascading changes to the underlying contracts would inevitably force direct modifications within the business logic. This top-down coupling directly violates the system's modularity constraints (NFR-6, NFR-7).

To decouple the algorithmic core from the storage and execution components, the system relies on the Dependency Inversion Principle. While modern dependency-inverting paradigms such as Clean Architecture \cite{martin2017clean} and Onion Architecture \cite{palermo2008onion} offer robust domain isolation, they prescribe concentric internal layers that mandate extensive \ac{DTO} mapping and rigid interface boundaries. For a localized command-line utility, operating under strict computational and latency budgets (NFR-1, NFR-3), this enterprise-level boilerplate introduces unjustifiable computational overhead. Furthermore, it inherently increases the structural complexity of the codebase, making future maintenance and the extension of new features unnecessarily burdensome for a utility application.

Therefore, this system adopts a \textbf{Hexagonal Architecture}, also known as the Ports and Adapters pattern \cite{cockburn2005hexagonal}. This pattern provides the necessary macro-level boundary control without dictating internal bureaucratic layers, allowing developers to maintain focus on the actual implementation of the core feature set. The architecture is divided into two strict conceptual zones:

\begin{itemize}
    \item \textbf{The Core (Inside):} This encapsulates the domain entities, the command validation (FR-6), and the mathematical normalization and score aggregation logic (NFR-8). The Core possesses zero knowledge of the external world, it dictates its storage and matching needs purely through abstract interfaces (Ports).
    \item \textbf{The Infrastructure (Outside):} This encompasses all external adapters that fulfill the Core's interfaces. This includes the user-interface, the storage backend, as well as the specific execution engines for the command option evaluation.
\end{itemize}
\FloatBarrier

By implementing interface ports at the absolute boundary of the Core, the external mechanisms -- whether they are the user interface, storage backend, or the command option evaluation mechanisms -- are forced to depend on the Core rather than the inverse. This guarantees that the core natural language orchestration and rank aggregation math can be compiled, tested, and executed entirely independently of the chosen storage backend or matching technologies, successfully fulfilling all modularity constraints.

\subsection{Static Architecture}

\subsubsection{System Context Model (Level 1)}
\label{sec:system_context}

To formalize the operational boundaries of the software and verify the isolation constraints, a Level 1 System Context diagram was modeled using the C4 standard \cite{brown2026c4}. As illustrated in Figure~\ref{fig:arch_level_1}, the proposed tool is treated as a unified, opaque system operating entirely within a headless Linux environment (ENV-4). This high-level abstraction defines the absolute perimeter of the application, explicitly identifying the external actors and systems permitted to interact with it. 

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.8\textwidth]{abbildungen/HLA-Diagram-Level-1.jpg} 
    \caption{Level 1 System Context Model}
    \label{fig:arch_level_1}
\end{figure}
\FloatBarrier

The system interacts with two primary driving actors that initiate operations:
\begin{itemize}
    \item \textbf{The End-User:} Interacts with the system by submitting natural language queries and configuration parameters (FR-1, FR-9). In turn, the End-User receives the command best fitting their query and disambiguation prompts (FR-7, FR-9).
    \item \textbf{The Catalog-Maintainer:} Represents either a human developer or an application developed by them. This actor interfaces with the system via an authenticated programmatic boundary to ingest new or updated, structured tool catalogs, receiving synchronous validation receipts upon success or failure (FR-3). The Catalog-Maintainer also has the ability to delete their catalogs using the authenticated interface.
\end{itemize}
\FloatBarrier

Furthermore, the system relies on a single driven external system: the \textbf{Host \ac{OS}}. The application initiates outward requests to the host operating system to read and write its persistent state. While the system manages application-level authentication, it defers to the underlying \ac{OS} file permission mechanisms to enforce \ac{OS}-level access control for all persisted data (NFR-11). 

Most critically, the Context Model serves as visual and structural proof of the system's privacy and offline execution limitations (ENV-2). It also demonstrates that the zero third-party daemon reliance (ENV-3) is considered. The defined perimeter strictly excludes any outbound network or third-party daemon dependencies. By restricting the attack surface and data flow strictly to the local host operating system and authorized standard inputs, the architecture guarantees that all user queries and proprietary tool descriptions remain entirely localized.

\subsubsection{System Context Model (Level 2)}
\label{sec:container_architecture}

Following the C4 modeling standard \cite{brown2026c4}, the Level 2 Container Diagram penetrates the opaque system boundary established in Level 1 to define the high-level, independently executable logical units. While the system is deployed to a localized, offline environment, it is structurally partitioned into two distinct containers. This separation formally illustrates the boundary between volatile computational execution and persistent state management, a necessary precursor to satisfying the storage modularity constraint (NFR-6).

As illustrated in Figure~\ref{fig:arch_level_2}, the system boundary encapsulates the following containers:

\begin{itemize}
    \item \textbf{The \ac{CLI} Executable:} A compiled, standalone binary (NFR-9) that contains the system's core logic. It acts as the sole active computational process.
    \item \textbf{The Local Database:} A daemonless data store strictly responsible for the persistent retention of system and configuration data. 
\end{itemize}
\FloatBarrier

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.8\textwidth]{abbildungen/HLA-Diagram-Level-2.jpg} 
    \caption{Level 2 Container Architecture}
    \label{fig:arch_level_2}
\end{figure}
\FloatBarrier

The inclusion of the system boundary (represented by the dashed perimeter) explicitly clarifies how external actors interface with the internal components. Both the \textbf{End-User} and the \textbf{Catalog-Maintainer} bypass the generic system perimeter to interact exclusively and directly with the \ac{CLI} Executable, submitting queries and ingestion payloads, respectively.

Crucially, the diagram illustrates the specific dependency mechanics of an embedded data persistence architecture. Because the persistence mechanism must be daemonless, it operates as an embedded library, hence the data persistence initiates no outward connections. The \ac{CLI} Executable maintains a strict, unidirectional dependency on the persistent storage, reading and writing state as needed. Consequently, it is the \ac{CLI} Executable--not the persistent storage itself--that initiates all direct interactions with the \textbf{Linux Host \ac{OS}} to delegate access control and execute the underlying file \ac{I/O} system calls.

\subsubsection{System Context Model (Level 3)}
\label{sec:component_architecture}

To detail the internal logical arrangement of the compiled application, the Level 3 Component Diagram decomposes the \ac{CLI} Executable container into its structural building blocks. It explicitly delineates the boundaries between the pure domain logic, the abstract interface contracts (Ports), and the pluggable execution infrastructure (Adapters).

As modeled in Figure~\ref{fig:arch_level_3}, the internal architecture strictly enforces the Dependency Inversion Principle. The components are categorized into three conceptual rings of execution:

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.8\textwidth]{abbildungen/HLA-Diagram-Level-3.jpg} 
    \caption{Level 3 Component Architecture detailing the Hexagonal boundaries}
    \label{fig:arch_level_3}
\end{figure}
\FloatBarrier

\paragraph{1. The Driving Adapters (Inbound Interface)}
Positioned at the perimeter, the driving adapters are responsible for receiving external actor input and translating it into domain-specific requests. 
\begin{itemize}
    \item The \textbf{\ac{CLI} Controller} parses standard input from the End-User, manages the interactive disambiguation loop (FR-7), and formats the output (FR-8).
    \item The \textbf{Ingestion API Handler} provides the authenticated boundary for receiving structured catalogs from the Catalog-Maintainer (FR-3). 
\end{itemize}
\FloatBarrier
Crucially, these adapters do not invoke the application core directly; rather, they depend on abstract inbound ports (\textbf{UserCommandPort} and \textbf{CatalogIngestionPort}).

\paragraph{2. The Isolated Core and Interface Ports}
The architectural center consists entirely of computational logic and the definition of the boundary interfaces.
\begin{itemize}
    \item \textbf{The Inbound Dependency Inversion:} The \textbf{Query Orchestrator} and the \textbf{Catalog Lifecycle Manager} serve as the primary use-case interactors. They implement the inbound ports. Consequently, their compile-time dependency arrows point inward toward the port interfaces, proving that the core dictates the execution contracts to the driving adapters.
    \item \textbf{Domain Mathematics \& Rules:} The \textbf{Similarity \& Rank Aggregator} executes the mathematics required for the unified ranking (NFR-8). The Query Orchestrator fetches definitions via the storage port and passes them in-memory to the \textbf{Syntactical Validator}. This allows the validator to evaluate command validity (FR-6) as a pure function without requiring its own \ac{I/O} capabilities.
\end{itemize}
\FloatBarrier

\paragraph{3. The Driven Adapters (Outbound Infrastructure)}
When the core requires external data or heavy computational execution, it calls its internally defined outbound interfaces (\texttt{StoragePort} and \texttt{MatchingStrategyPort}). The driven adapters exist outside the core to physically fulfill these requests.
\begin{itemize}
    \item The \textbf{Persistence Adapter} implements the \textit{StoragePort} interface, translating domain entities into the structures required by the underlying storage mechanism and executing the underlying host \ac{OS} file operations (NFR-6, NFR-11).
    \item The \textbf{Matching Strategy Adapters [1..*]} fulfill the matching interface. By utilizing the extension \texttt{1..n}, the diagram shows that the core is entirely agnostic to the number or type of search algorithms employed (NFR-7). 
\end{itemize}
\FloatBarrier

Ultimately, this Level 3 configuration guarantees that all dependencies point inward toward the abstract ports. Because the core components only depend on interfaces they themselves define, the underlying storage mechanisms or scoring mechanisms can be entirely replaced without modifying the central orchestration or mathematical logic.

\subsubsection{Omission of C4 Level 4}

While the C4 framework includes a fourth level for mapping individual components to code-level constructs, it is treated as an optional level of detail within this specification. As noted by the C4 standard, this depth is typically reserved for only the most complex components \cite{brown2026c4}. Because class structures are highly volatile and often available on-demand via tooling options\cite{brown2026c4}, documenting them here would risk creating brittle artifacts. Instead, the architecture relies on the Level 3 component boundaries to provide sufficient rigor, leaving low-level attributes and methods to be automatically generated where deemed necessary.

\subsection{Dynamic Architecture (System Behavior)}
\label{sec:dynamic_architecture}

While the static architecture defined in Section~\ref{sec:component_architecture} establishes the strict structural boundaries of the system, it does not describe the runtime execution logic. This section models the dynamic behavior of the \ac{CLI} application using \ac{BPMN} 2.0 standard notation. To prevent visual bloating, some diagrams have been structured hierarchically by adding subprocess diagrams.

\paragraph{Architectural Execution Model (The Stateless Lifecycle)}
\label{sec:execution_model}

To strictly satisfy the \enquote{Single Linux Deployment Artifact} (NFR-9) and \enquote{Zero Third-Party Daemon} (ENV-3) constraints, the system is designed as a purely stateless executable. The application does not rely on background listeners or idle memory allocation. Instead, every interaction follows a strictly bounded lifecycle: the process is initialized by the host \ac{OS}, allocates memory Just-In-Time (JIT), executes the requested domain logic, and immediately terminates. 

This daemonless approach guarantees that \ac{RAM} limits (NFR-3) are only consumed during active execution and that only one executable has to be installed.

\label{sec:primary_query_flow}

The primary functional purpose of the system is to translate natural language into executable terminal commands (FR-1). Figure~\ref{fig:bpmn_high_level} illustrates the high-level system overview of this interaction.

If the highest resulting score remains below the minimum confidence threshold, the process immediately aborts evaluation and returns a \enquote{Tool Undetermined} state.

\paragraph{The Interactive Disambiguation Loop}
If the Tool and Option Determination subprocess fails to achieve the required confidence threshold, or if the Command Verification subprocess encounters a validation error, an interactive disambiguation loop (FR-7) is triggered. The system issues a clarification request to the End-User, and waits for a rewritten query. Crucially, this loop routes back to the tokenization phase, explicitly bypassing the engine initialization phase to minimize latency on subsequent iterations.

\label{sec:administrative_flows}

Beyond query resolution, the system must securely handle state mutations initiated by both users and external maintainer applications.

\paragraph{Catalog Lifecycle Management}

Once authorized, an exclusive gateway routes the payload to either the ingestion, update or deletion logic. If a catalog is being created or updated, the system preprocesses the ingested catalog data before persisting it to the data store, to ensure it fits the format required by the application. Regardless of the action, the system guarantees a success or failure receipt before the process terminates, ensuring maintainers can rely on a status message.

\subsection{Interface Specifications}
\label{sec:interfaces}

To finalize the systems design, it is necessary to define the data contracts that bridge the system's boundaries. This section is explicitly divided into two categories: Internal Interfaces and External Interfaces. This distinguishes programmatic domain contracts (inside the Hexagon) from the interaction pathways used by actors (outside the Hexagon).

\subsubsection{Internal Interfaces (Hexagonal Ports)}
\label{sec:internal_interfaces}

The internal interfaces formalize the Inbound (Driving) and Outbound (Driven) ports introduced during the Level 3 Component Architecture breakdown (Section~\ref{sec:component_architecture}).

The programmatic contracts -- comprising the required method signatures, input parameters, return types, and exceptions -- are explicitly outlined in the technical tables provided in Appendix~\ref{app:interfaces}. Defining these parameters formally concludes the design of the application core, guaranteeing its total isolation from external adapters.

\subsection{Technology Decisions}

After establishing the architectural design of the system this section discusses the technology decisions considered for this project. Because the programming language forms the basis of the system, the section begins by evaluating and picking a suitable programming language for the project. It then compares different ways of implementing the persistence layer of the tool before discussing which matching engines the system should utilize to determine the correct commands.

\subsubsection{Matching Engines}\label{sec:engines}

The literature review identified two approaches that might be used as matching engines. The first of them was the use of \acp{LLM}. 

\paragraph{\acp{LLM}} process natural language and tools like ChatGPT or Gemini are known to be able to generate shell commands from this input. Because the tool follows a local-first approach the project would, however, not be able to utilize any cloud-hosted \acp{LLM} in this project. The project would therefore be limited to open-weight models that are able to run on local hardware. Unfortunately even small one-billion-parameter models take up a significant amount of memory. Even if quantized down to 8-bit a model would use up about 1000MB of memory, for its weights alone (1 Parameter = 8 bit = 1 byte), making it scratch the memory limitation of NFR-3 and also leading to an increased startup and processing time (NFR-1, NFR-2). Only a 4-bit quantized version could fit comfortably into the constraints of NFR-3. Such a model, would however suffer from significant performance penalties~\cite{lee2024exploring}. Furthermore, the model's knowledge would be confined to the data it was trained on. To effectively provide it with novel data a \ac{RAG}-Architecture would have to be implemented. This would enable the system to retrieve data about new tools and augment the natural language prompt with it~\cite{lewis2020retrieval}. However, that approach would require an additional pipeline, which itself may cause significant computational overhead, making it not feasible. 

\paragraph{Specialized Neural Networks} were identified as the second approach. This has previously proven successful to a certain degree~\cite{agarwal2021neurips}. The major constraint however, as with a traditional \acp{LLM}, would be the limitation to the training data~\cite{10488711}. In practice, supporting new or proprietary commands would therefore require collecting additional labeled examples and retraining, which conflicts with the projects' goal of an extensible system where commands can be added without complex model updates. Additionally, training and maintaining a specialized neural network for a large and evolving command space can require non-trivial data collection and compute resources~\cite{thompson2020computational}, particularly if the model must be periodically updated to reflect new commands and environments. It was therefore decided that it may be a good supplementary engine to be explored in future work, to improve the retrieval accuracy of publicly documented commands. This study however, focuses on identifying an approach that will also allow for commands that are not publicly available to be retrieved. Hence, this approach will not be explored any further in this work.

By exploring the \ac{RAG} approach, to extend the functionality of \acp{LLM}, the study identified two other methods for retrieving command line tools and their options~\cite{aljohani2026enhancing,baban2025optimizing}. 

\paragraph{Embedding Models} being the first one of them. Instead of directly translating a user query into a full command and option response, this approach calculates a vector representation of the user query. If vector representations are also available for a vector representation of the tools and their options, cosine similarity can be used to identify the closest semantic match to the query. Since a selection exists of embedding models using significantly fewer parameters than one billion, the memory and latency constraints are less restrictive here. Additionally, the corpus of commands and their options would be easily extensible by just calculating their vector representations and adding them. This would allow for new commands and options to be added at any time and therefore not just be limited to the training data. For these reasons it was decided to include an embedding model as one of the matching engines. Chapter~\ref{sec:comp_emb_models} will identify and compare a selection of embedding models, to identify the best possible candidate for this system.

\paragraph{Keyword Search} was identified as the second method. This approach would be highly efficient, using up only a fraction of the 1024 MB memory budget~\cite{li2026context}. It is also highly efficient in terms of required processing power~\cite{santhanam2023moving}, making it suitable to supplement the retrieval results of the embedding models. It is also easily extensible, all that needs to be done is add a new catalog containing descriptions, keywords or both for the new tool and its options. A simple keyword search was therefore selected as the secondary engine. As it is the de facto standard for such tasks, it was decided to use the BM25 algorithm~\cite{jeong2026bayesian}.

\subsubsection{Programming Language}

\paragraph{Python} is frequently selected as the language for machine learning projects **source**. It is established as the de facto standard, uses simple syntax and has a variety of frameworks for such projects available **sources**. However, those frameworks come with a big performance and memory footprint **source** and while a lot of the frameworks are actually written in C or C++ **source**, native Python is considered quite slow **sources**. This makes it not suitable for the project due to the NFR-1 and NFR-3 constraints.

\paragraph{C++} is the language behind the llama-cpp library, which can be used to run  embedding models **source**. It is also considered to be a fast programming language **source** since unlike Python, it does not require an interpreter to run, which also leads to a lower memory footprint. This makes it a reasonable choice for the project. However, it was decided that due to the lack of memory safety **source** and missing experience in the use of the language by the team, C++ is not the right fit for the project.

\paragraph{Rust} is a compiled language that is considered memory safe **source**. It therefore combines the speed and memory efficiency of an interpreterless language like C++ with the feature of memory safety. Additionally, the llama-cpp library is also available for Rust which enables their use for running embedding models. All this in addition to the team being experienced in the use of the language makes Rust the best fit for the project. Hence, Rust will be used as the programming language in the implementation phase.

\subsubsection{Persistence Layer}

The environmental constraint ENV-3 strictly requires the application to not use any third-party daemons. Due to this, traditional \acp{DBMS} cannot be used to persist the tool catalogs. This leaves two available options:

\paragraph{Structured files} would allow data to be stored directly in a file. No third-party daemon would be required and by utilizing an established structured format like JSON or XML, no custom parser would have to be developed for accessing any of the data. However, this approach has two significant drawbacks. The first one being, that despite parsers existing for such files, the data retrieval itself would be very slow compared to an indexed database **source**. This might be an issue especially in a big corpus of data. On the other hand the bigger of the drawbacks, is that structured files do not support optimized vector retrieval. This would lead to poor query performance and thereby endanger the 1000 ms query latency limit specified by NFR-1.

\paragraph{SQLite} is a file-based database system. Unlike traditional \ac{DBMS} it does not require a database server running in the background. The database engine is contained within the connection library itself **source**, hence it gets compiled directly into the application, circumventing the ENV-3 constraint of no third-party daemons because there is no daemon running. Despite this architectural choice, SQLite functions much like a traditional database. It allows for indexing **source** and guarantees ACID properties. While native SQLite does not support any optimizations for the retrieval of vectors, extensions can be added to achieve this **source**. The sqlite-vec library for example allows for a direct cosine distance check between stored vectors and a target vector **source**. It was therefore decided that SQLite in combination with the sqlite-vec extension is the most suitable persistence layer for this project.
